# ERP合并功能集成设计文档
生成时间：2026年07月16日

## 需求背景
把独立的 `ERP_merge` 项目最新版本功能集成到 `work_order_process` 项目中，作为子命令统一管理，同时满足以下约束：
1. 保留原独立脚本的运行能力，同时新增子命令入口
2. 统一Python版本为 `>=3.14`
3. 源文件、规范文档纳入项目规范管理
4. 所有规则配置化，使用TOML格式，和项目现有风格统一
5. 默认不输出结果Excel文件（数据量过大），仅保留处理统计日志，需要输出时通过参数主动开启
6. 仅集成最新版本 `merge_erp_data_20260715.py` 的功能，旧版本脚本和对应测试清理

## 整体架构与目录结构
### 1. 目录树
```text
work_order_process/
├── src/work_order_process/
│   ├── ...（原有工单模块）
│   └── erp_merge/  ← 新增ERP合并模块
│       ├── __init__.py
│       ├── cli.py  ← 子命令入口，参数解析、日志输出
│       ├── config.py  ← TOML规则配置加载与校验
│       ├── mapping.py  ← 字段映射、营销平台映射、体系工程师匹配、金额换算逻辑
│       ├── calculator.py  ← 合同天数、年度分摊服务费计算核心逻辑
│       └── io.py  ← Excel文件读写、输入输出路径处理
├── config/
│   └── erp_merge_rules.toml  ← 全规则配置文件
├── data/erp_merge/  ← 源数据Excel存放目录
├── docs/erp_merge/  ← ERP合并规则文档目录
├── tests/erp_merge/  ← 测试目录
│   ├── test_mapping.py
│   ├── test_calculator.py
│   └── test_cli.py
├── merge_erp_data_20260715.py  ← 保留的独立运行入口，内部调用erp_merge模块
├── pyproject.toml  ← 新增依赖、注册子命令入口
└── ...（原有文件）
```
### 2. 模块职责
- 核心逻辑全在 `erp_merge` 包内，和原有工单功能完全隔离，互不影响
- 双入口最终都调用包内同一套核心函数，逻辑完全一致，不会出现行为差异
- 旧版本脚本和对应测试文件完全清理，无冗余代码

### 2. 双入口实现逻辑
- **子命令入口** `work_order_process erp-merge`：在 `pyproject.toml` 中注册，调用 `erp_merge.cli.main()`，参数风格和原有工单命令统一，支持自定义配置文件路径、输入输出路径、统计日期区间覆盖
- **独立脚本入口** `merge_erp_data_20260715.py`：仅保留 5 行以内的薄包装，逻辑改为 `from work_order_process.erp_merge.cli import main; main()`，不包含任何业务代码，独自运行行为和子命令完全一致
- 两者共用同一套核心逻辑、同一份配置文件，输出结果不会出现差异
### 3. 数据流与异常处理
- 输入：默认读取 `data/erp_merge` 目录下的新旧ERP源Excel，也支持命令行参数自定义路径
- 输出：正常运行时结果Excel默认输出到 `output/erp_merge` 目录，同步输出处理统计日志到控制台；本次因数据量过大仅做测试运行，不输出最终结果文件，后续正式使用时恢复默认输出行为，也支持 `--no-output` 参数主动关闭结果文件输出
- 异常：配置文件字段缺失、源文件不存在、关键字段为空等场景有明确中文错误提示，风格与原有工单命令统一
- 依赖：在 `pyproject.toml` 中新增 `pandas>=2.2.0`/`numpy>=1.26.0`/`openpyxl>=3.1.5` 三个运行时依赖，Python版本统一为 `>=3.14`
